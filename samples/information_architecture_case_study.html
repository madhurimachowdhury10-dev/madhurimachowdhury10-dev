Case Study · Information Architecture

Designing an Information Architecture That Scales

How a fragmented SaaS knowledge base was transformed into a task-driven self-service experience focused on discoverability, scalability, and governance.

When Documentation Starts Working Against Users

The knowledge base wasn't technically broken. Users could still find answers. Eventually.

Over time, content had grown faster than the structure supporting it. New articles were added by different teams, categories expanded organically, and navigation increasingly reflected internal ownership rather than user goals.

📂 Duplicate Paths

Similar topics appeared in multiple categories.

🔎 Search Dependency

Users relied on search instead of navigation.

🧭 Team-Based Structure

Navigation reflected ownership, not tasks.

📈 Growth Without Governance

No framework existed for adding new content.

Understanding User Intent

The existing architecture was organized around product ownership. User behavior suggested it should be organized around tasks.

What Users Want

├── Get Started
├── Manage Users
├── Connect Integrations
├── Fix Problems
└── Manage Billing

Mapping the Existing Experience

Knowledge Base

├── Account
├── Admin
├── Users
├── Settings
├── Configuration
├── Troubleshooting
├── Integrations
├── Billing
└── Security
Key findings:
  • User permissions appeared in three different sections.
  • Billing content existed under both Account and Billing.
  • Troubleshooting articles were scattered across categories.

Designing a Task-Based Architecture

│
└── Security & Compliance

🎯 Task-Based Navigation

Categories organized around user goals rather than internal teams.

📚 Consistent Taxonomy

Standardized naming and placement rules.

📈 Scalable Growth

Future content can be added without restructuring.

Key Takeaways

User goals should drive structure.
Taxonomy is a product decision.
Governance matters as much as navigation.
Scalability must be designed intentionally.
